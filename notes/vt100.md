# VT100 / VT220 Terminals

> Learning note. Facts cite source; assumptions are marked _(assumption)_.

## Question
What does 'VT100 compatible' actually mean today, and what is the Williams parser
every modern terminal is built around?

## Key concepts
- DEC VT hardware lineage
- the Paul Williams DEC ANSI-compatible parser state machine
- modes (DECSET/DECRST), character sets
- why emulators still target VT100/VT220

## Findings

### The Williams parser (the shared skeleton)
Modern terminals model the escape-sequence grammar as the state machine Paul
Williams reverse-engineered from DEC hardware
(vt100.net/emu/dec_ansi_parser). Verified: Ghostty, WezTerm, and tmux all cite
it directly (`comparisons/parser.md`).

**States:** `GROUND, ESCAPE, ESCAPE_INTERMEDIATE, CSI_ENTRY, CSI_PARAM,
CSI_INTERMEDIATE, CSI_IGNORE, DCS_ENTRY, DCS_PARAM, DCS_INTERMEDIATE,
DCS_PASSTHROUGH, DCS_IGNORE, OSC_STRING, SOS_PM_APC_STRING`.

**Actions:** `ignore, print, execute, clear, collect, param, esc_dispatch,
csi_dispatch, hook, put, unhook, osc_start, osc_put, osc_end`.

**Three-phase transitions:** a transition can carry an **exit action** (leaving
the old state), a **transition action**, and an **entry action** (into the new
state), run in that order. Ghostty implements exactly this triple
(`projects/ghostty/src/terminal/Parser.zig:262-310`).

### "VT100 compatible" today
_(interpretation)_ It means the terminal accepts the DEC VT family's escape
grammar — the C0/C1 controls, CSI/OSC/DCS structure, and the DEC private modes
(DECSET/DECRST via `CSI ? … h/l`, e.g. `?25` cursor visibility, `?1049` alt
screen, `?2004` bracketed paste). The `TERM`/terminfo entry a terminal ships
(`xterm-256color`, `xterm-ghostty`, `xterm-kitty`) advertises which capabilities
programs may use — see `pty.md`.

### UTF-8 is bolted on
The classic diagram predates UTF-8; each terminal extends it differently
(Ghostty SIMD decode-until-control; WezTerm dedicated `Utf8`/`Utf8Sequence`
state; Kitty in-loop bulk decode; tmux its own collector). _(assumption)_ so
codepoint decoding is *not* part of the canonical Williams states.
